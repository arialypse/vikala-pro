# Vikala Pro - Developer Handover & Reference Document

เอกสารฉบับนี้คือ **"Reference Guide"** ที่สรุปประเด็นสำคัญ ข้อตกลง สถาปัตยกรรมระบบ และคำแนะนำเชิงลึกทั้งหมดจากห้องสนทนานี้ เพื่อส่งมอบต่อให้คุณ (AI Developer ในห้องถัดไป) ใช้เป็นบริบทหลักในการเริ่มเขียนโค้ดสำหรับโปรเจกต์แอปพลิเคชัน **"Vikala Pro"** (โปรแกรมออกใบเสร็จรับเงิน/ใบกำกับภาษีระบบ Multi-Company) ควบคู่กับไฟล์พิมพ์เขียว `Vikala-Pro-design.md` และไฟล์เทมเพลตหน้าจอ `WealthWise-v2-index.html`

## 1. บริบทสภาพแวดล้อมระบบและข้อจำกัดวิกฤต (Critical Constraints)

โปรแกรมเมอร์ AI ต้องยึดถือสภาพแวดล้อมนี้เป็นตัวตั้งต้นในการเขียนโค้ดเสมอเพื่อไม่ให้ระบบพังเมื่อขึ้นระบบจริง (Production):
* **Infrastructure:** รันบนระบบแชร์โฮสติ้ง **cPanel เวอร์ชัน 134.0.30** มีสิทธิ์เข้าถึงผ่าน FTP เท่านั้น (ไม่มี SSH Access)
* **Backend & DB:** ภาษา **PHP 8.2.31** และระบบฐานข้อมูล **10.6.27-MariaDB (mysqlnd 8.2.31)**
* **⚠️ ข้อจำกัดเรื่องพื้นที่ดิสก์ (Disk Space Alert):** สถานะฮาร์ดดิสก์หลักของเซิร์ฟเวอร์ (Disk /) มีอัตราการใช้งานสูงถึง **89%** * *กฎเหล็ก:* **ห้ามเขียนโค้ดให้บันทึกไฟล์ PDF ลงบนเซิร์ฟเวอร์ดิสก์เด็ดขาด** * *แนวทาง:* ให้เขียนตรรกะประมวลผลและสร้างไฟล์ PDF (ใบเสร็จ/ใบกำกับภาษี) แบบ **Real-time ในหน่วยความจำ (In-Memory)** เมื่อผู้ใช้กดคลิกพิมพ์หรือดาวน์โหลดเท่านั้น เพื่อป้องกันปัญหาดิสก์เต็ม

## 2. ทิศทางการเลือกใช้เครื่องมือและการเขียนโค้ด (Tech Stack Directives)

* **PHP Framework:** บังคับใช้ **CodeIgniter 4 (CI4)** เสมอ
    * *เหตุผล:* เป็นมิตรกับ Shared Hosting (cPanel) สูงสุด สามารถติดตั้งและอัปเดตโครงสร้างไฟล์ได้ง่ายๆ ผ่าน FTP โดยไม่ต้องรันคอมมานด์ไลน์ที่ยุ่งยาก มีสเปคที่เบาแต่มีระบบ Security (CSRF, SQL Injection) ติดมาพร้อมใช้งาน
* **PDF Library:** แนะนำให้ใช้ **mPDF** หรือ **Dompdf** ที่เสถียรกับการประมวลผลภาษาไทยและรองรับการจัดหน้ากระดาษแบบ A4

## 3. สถาปัตยกรรม UI/UX และระบบแม่แบบเอกสาร (Design & Print Mechanics)

โครงสร้างหน้าตาของโปรแกรมจะแบ่งออกเป็น 2 ส่วนที่ใช้กลไกต่างกันโดยสิ้นเชิง:
1.  **หน้าจอแอปพลิเคชันเว็บ (Web Dashboard):** ดัดแปลงโครงสร้างและคลาสสีจากไฟล์ **`WealthWise-v2-index.html`** (ใช้โทนสีคราม Indigo/Slate) และต้องรองรับระบบ **Dark/Light Mode Theme** สำหรับหน้าจอตัวพรีวิวบนเว็บด้วย
2.  **ระบบพิมพ์เอกสารลงกระดาษ/ดาวน์โหลด PDF:**
    * *กฎเหล็ก:* **ห้ามใช้ CSS Flexbox หรือ CSS Grid ในไฟล์หน้าเทมเพลตพิมพ์เด็ดขาด** เนื่องจากเอนจินของไลบรารี PDF หลังบ้าน (เช่น mPDF) ไม่รองรับ จะทำให้เลย์เอาต์หน้ากระดาษพังและข้อความซ้อนกัน 
    * *แนวทาง:* บังคับให้ใช้โครงสร้างตารางแบบดั้งเดิม (**HTML Table Structure เช่น `<table>`, `<tr>`, `<td>`**) ในการจัดเลย์เอาต์ซ้าย-ขวา เพื่อความแม่นยำ 100% บนหน้ากระดาษ A4
    * *การประหยัดหมึกพิมพ์ (Ink-Saving Mode):* บังคับเขียน `@media print` เพื่อล้างค่าสีพื้นหลังทึบทั้งหมดออกเมื่อสั่งพิมพ์ ให้เหลือเพียงเส้นกรอบคั่นบางๆ (Minimalist Style) ข้อความสีขาวบนหน้าเว็บต้องถูกบังคับเปลี่ยนเป็นสีดำสนิทเมื่อพิมพ์ เพื่อความประหยัดและคมชัดตามมาตรฐานงานบัญชี

## 4. ตรรกะทางธุรกิจที่ห้ามตกหล่น (Core Business Logic for Developer)

เมื่อเริ่มลงมือเขียนระบบฐานข้อมูลและ Controller โปรดตรวจสอบให้แน่ใจว่าได้ระบุกฎเกณฑ์เหล่านี้ลงไปในโค้ด:

### A. การทำ Data Snapshot ป้องกันข้อมูลเปลี่ยนรูป (Critical)
* *ปัญหาที่ต้องแก้:* ฐานข้อมูลลูกค้า (`customers`) เป็นส่วนกลางที่แชร์ให้ทุกบริษัทดึงไปใช้ หากลูกค้าขอย้ายที่อยู่หรือเปลี่ยนชื่อในอนาคต เอกสารในอดีตที่เคยออกไปแล้วจะเปลี่ยนตาม ซึ่งผิดกฎหมายบัญชีอย่างร้ายแรง
* *วิธีเขียนโค้ด:* ในจังหวะที่ User กดปุ่มบันทึกเอกสาร (Invoice/Receipt/TaxInvoice) ตรรกะใน Controller จะต้องดึงข้อมูล ชื่อ, ที่อยู่, เลขผู้เสียภาษี และรหัสสาขาของลูกค้า ณ วินาทีนั้น มาลงคีย์ **เซฟฝังตัว (Snapshot)** ลงในคอลัมน์ `customer_name`, `customer_address`, `customer_tax_id`, `customer_branch` ของตาราง `documents` โดยตรงเพื่อล็อกข้อมูลในอดีตให้คงที่ตลอดไป

### B. ข้อกำหนดภาษีสรรพากรไทย (Thai Tax Compliance)
* **รหัสสาขา 5 หลัก:** ระบบต้องบังคับให้มีรหัสสาขา 5 หลักเสมอ โดยกำหนดค่าเริ่มต้นเป็น `00000` สำหรับสำนักงานใหญ่ และรหัสอื่นๆ สำหรับสาขา เพื่อให้ใบกำกับภาษีถูกต้องตามกฎหมาย
* **ฟิลด์แยกมูลค่าภาษี (VAT Tracking):** ในตาราง `documents` บังคับให้คำนวณและบันทึกค่า `vat_rate` และ `vat_amount` แยกออกมาทุกครั้ง เพื่อนำตัวเลขชุดนี้ไปออกรายงานภาษีขายประจำเดือนส่งสรรพากรได้อย่างสมบูรณ์

### C. การป้องกันข้อมูลพังและการซ่อนข้อมูล (Data Integrity & Soft Inactivation)
* **ห้ามลบข้อมูลคู่ค้าและสินค้าเด็ดขาด:** เพื่อป้องกันไม่ให้ประวัติเอกสารบัญชีเก่าๆ พังเมื่อไอเทมถูกลบ 
* **วิธีเขียนโค้ด:** ระบบจะใช้ฟิลด์สถานะ `is_active` (TINYINT 1) เป็นตัวควบคุมในตาราง `customers` และ `products` หากข้อมูลไหนเลิกใช้งาน ให้ผู้ใช้กดปิดสถานะ (เป็น 0) ระบบจะซ่อนข้อมูลนั้นออกจากการแสดงผลใน Dropdown ตอนสร้างเอกสารใหม่ทันที โดยที่ข้อมูลดิบในอดีตยังปลอดภัย 100%

### D. การรองรับบิลมูลค่าสูงและการเชื่อมโยงเอกสาร
* **ตัวเลขอสังหาริมทรัพย์ระดับพันล้าน:** ฟิลด์การเงินและภาษีทั้งหมดในตาราง `documents` และ `document_items` ได้ถูกปรับแต่งสเปคให้ใช้ **`DECIMAL(12,2)`** เพื่อป้องกันระบบ Error (Out of range) เมื่อบริษัทออกบิลโครงการใหญ่ที่มีมูลค่ารวมสูงสุดถึงระดับเก้าพันล้านบาท
* **เลขที่อ้างอิงเวิร์กโฟลว์ (`reference_number`):** ในโครงสร้างตารางเอกสาร มีฟิลด์นี้เตรียมไว้เพื่อให้คุณเขียนโค้ดผูกความสัมพันธ์เชื่อมโยง เช่น เมื่อออกใบเสร็จรับเงิน (Receipt) สามารถดึงรหัสเลขที่ใบแจ้งหนี้เดิม (Invoice) มาบันทึกผูกโยงเพื่อตรวจเช็กสถานะการเก็บเงินได้

### E. ระบบสวิตช์เปิด-ปิดสไตล์จาก Setting ส่วนกลาง
* ในตารางบริษัท (`companies`) จะมีฟิลด์ `show_remarks` และ `show_signatures` เพื่อให้ Admin ตั้งค่าไว้ครั้งเดียวจากเมนูตั้งค่าส่วนกลาง โค้ดในหน้า View เอกสารจะต้องอ่านค่าจากฟิลด์นี้มาทำเงื่อนไข `if/else` เพื่อเปิดหรือซ่อนข้อความหมายเหตุชำระเงินและกล่องเซ็นลายมือชื่อด้านล่างกระดาษโดยอัตโนมัติ ไม่ต้องให้พนักงานติ๊กเลือกใหม่ในทุกๆ เอกสาร
